1. **導入**: LLMエージェントによるセキュリティ監査の可能性

### スライド 3: 複数実装が「同じ誤解」を共有すると既存手法では検出できない

- スライド上:
  - 図: 3つの実装A, B, Cが同じ仕様を参照し、同じ誤解をする様子
  - 差分テストの矢印: A↔B、B↔C、A↔C すべて「差分なし」
  - 赤字: **意味的盲点**（セマンティック・ブラインドスポット）
  - 引用: Knight & Leveson 1986「独立実装でも相関故障は起きる」

- 話す内容:
  差分ファジングは複数実装の挙動差をオラクルとして活用する強力な手法です。しかし、根本的な限界があります。すべての実装が同じ仕様の曖昧さを同じ方向に誤解していた場合、挙動の差分が出ないため、バグは見逃されます。私たちはこれを「意味的盲点」と呼びます。1986年のKnight & Levesonの研究でも、独立に開発された実装が共通して失敗する現象が報告されています。（40秒）

- 次への接続:
  この盲点に対処するのが、本研究の核心的アイデアです。

### スライド 4: 仕様からチェックリストを抽出し横展開することで盲点に対処する

- スライド上:

  - 左: 仕様書（MUST/SHOULD/MAY）
  - 右: 複数実装への矢印（1つの発見→他実装へ横展開）
  - キーワード: **仕様チェックリスト化** / **横断実装チェック**

- 話す内容:
  本研究の核心は、仕様の規範要件をチェックリストとして固定し、それを複数実装に適用することです。差分テストが「実装同士の比較」に依存するのに対し、私たちは「仕様を真実の源泉」として監査します。さらに、1つの実装で見つけた問題パターンを他実装に横展開する「横断実装チェック」により、監査を1→Nにスケールさせます。（35秒）

### スライド 5: SpecAuditは2フェーズ・3戦略で監査を体系化する

- スライド上:
  - アーキテクチャ図:
    ```
    フェーズ1: 知識構造化
    ┌─────────┐   ┌──────────┐   ┌──────────┐
    │ 仕様書  │──▶│規範要件  │──▶│パターンDB│
    │(自然言語)│   │ 抽出     │   │ (50+)    │
    └─────────┘   └──────────┘   └──────────┘
                        │
                        ▼
    フェーズ2: 体系的監査
    ┌────────────────────────────────────┐
    │ 戦略A: 仕様ベース静的監査          │
    │ 戦略B: 横断実装チェック            │
    │ 戦略C: 動的テスト生成              │
    └────────────────────────────────────┘
    ```
  - 成果物: 規範要件ID, 実装マッピング, チェックリスト, 監査レポート

- 話す内容:
  SpecAuditは2つのフェーズで構成されます。フェーズ1では自然言語仕様から規範要件を抽出し、実装コードへのマッピングを構築します。フェーズ2では3つの戦略で監査を実行します。戦略Aは仕様に基づく直接検証、戦略Bは発見パターンの横展開、戦略Cは動的テスト生成です。すべての工程で検査可能な成果物を生成し、再現性を担保します。（40秒）

- 次への接続:
  特に重要なのが戦略B、横断実装チェックです。

### スライド 6: 横断実装チェックは1つの発見を複数実装に展開する【見せ場B】

- スライド上:
  - フロー図:
    ```
    クライアントAで発見    パターン抽象化      他実装をチェック
    ┌─────────────┐      ┌─────────────┐      ┌─────────────┐
    │スロットガード│─────▶│範囲ガード   │─────▶│クライアントB: ✗│
    │欠落を発見   │      │欠落パターン │      │クライアントC: ✗│
    └─────────────┘      └─────────────┘      │クライアントD: ✓│
                                              └─────────────┘
    ```
  - 具体例テーブル:
    | 元の発見 | パターン | 横展開結果 |
    |---------|---------|-----------|
    | クライアントA: スロットガード欠落 | 範囲ガード欠落 | B,C同様に欠落 |
    | クライアントC: バッチ境界不整合 | 配列境界チェック | Aでも発見 |

- 話す内容:
  横断実装チェックの仕組みを説明します。まず、ある実装で問題を発見したら、その問題を抽象的なパターンとして一般化します。例えば「最古スロットガードの欠落」は「範囲ガード欠落パターン」になります。次に、このパターンを他の全実装で検索します。仕様の曖昧な部分は複数チームが同様に誤解しやすいため、1つの発見が複数の発見に繋がります。（45秒）

- 次への接続:
  この手法を実運用環境で評価しました。
